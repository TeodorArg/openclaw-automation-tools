# Release Records

This repo keeps durable package release records under `docs/releases/`.

Release storage contract:

- use one package directory per live publishable plugin package
- keep a package index at `docs/releases/<package-slug>/README.md`
- keep one small Markdown note per released version at `docs/releases/<package-slug>/vX.Y.Z.md`
- use that per-version file as the canonical operator copy-paste source for GitHub Release text and the tracked release record for the package version
- keep a companion worksheet file at `docs/releases/<package-slug>/vX.Y.Z.clawhub.md` for every plugin release
- keep a fresh release archive in the package directory for every plugin release, using the package-qualified tarball built from the release state
- use repo-level canonical scaffolds `docs/releases/TEMPLATE.md` and `docs/releases/TEMPLATE_CLAWHUB.md` when a package does not carry a narrower local template
- package-local templates such as `TEMPLATE.md` and `TEMPLATE_CLAWHUB.md` may already exist, but they do not replace generating both versioned files for each release
- keep transient release prep checklists in `.local-planning/`, not in `docs/releases/`
- use package-qualified tags in the form `<package-slug>/vX.Y.Z`

Release boundary:

- a merged PR is not a package release by itself
- for this repo, the default public release centers on the package-qualified tag plus GitHub Release, and the release slice is not closed until the package version is aligned, the merge is on `main`, the package-qualified tag is pushed, the GitHub Release is published from that tag, verification is green, the fresh release archive exists in the package directory, and both tracked release artifacts are backfilled
- do not treat the release task as closed until the tracked `vX.Y.Z.md` and `vX.Y.Z.clawhub.md` files both carry the real GitHub Release URL, required tag-backed source fields such as `Source commit` and `Source ref`, and the fresh release archive path
- ClawHub publish or manual archive upload is optional by default and remains a separate operator step unless explicitly requested for that release
- GitHub Releases are public release records for this repo even though ClawHub itself does not require them for package publication

Required fields for each `docs/releases/<package-slug>/vX.Y.Z.md`:

- package slug
- package version
- release date
- tag name
- tag commit SHA
- GitHub Release URL
- whether GitHub autogenerated notes were mirrored or supplemented
- short summary
- marketing description review status
- changelog quality review status
- verification checklist
- archive path
- source commit
- source ref
- publish result
- companion worksheet path
- user-facing version notes that describe the shipped change set concretely and not just the package purpose

Required fields for each `docs/releases/<package-slug>/vX.Y.Z.clawhub.md` worksheet:

- package slug
- package version
- package type
- package name
- display name
- owner or publisher
- changelog text
- source repo
- source commit
- source ref
- source path
- tarball file

GitHub release notes policy:

- the tracked `docs/releases/<package-slug>/vX.Y.Z.md` file is the canonical curated release note artifact in this repo
- GitHub autogenerated release notes may supplement that file and are configured through `.github/release.yml`
- if autogenerated notes are used in GitHub, mirror that fact or the curated summary back into the tracked file so the repo keeps one durable source
- the curated summary must describe the shipped version-specific changes and not fall back to generic package-purpose copy
- the tracked note should usually contain both a short summary and a more detailed shipped-change section with concrete version-specific bullets
- create the GitHub Release only after the tracked note is in a public-ready state, and if the tracked note is improved after publication, update the GitHub Release body from that file

ClawHub changelog policy:

- the worksheet `Changelog text` is not just a one-line mirror of the package summary; it should be a compact listing-facing changelog for the shipped version
- keep it shorter than the GitHub Release notes, but still concrete: usually 2-4 concise lines or bullets naming the shipped changes
- avoid mechanical wording like `updates metadata`, `republishes the package`, or `keeps things aligned` unless that is the true release scope
