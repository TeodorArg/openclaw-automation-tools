# Canon Ops Orchestrator

Use this bundled skill to choose the right operating mode for canon work.

## Operating modes

Use one of these fixed modes:
- `status`
- `doctor`
- `preview-fix`
- `apply-safe-fix`
- `source-preflight`

## Local first

Use `canon_status` locally when:
- you need the latest summary only
- you are checking whether canon work is needed at all
- you want a lightweight refresh before planning
- you do not intend to preview or apply fixes in the same pass

Treat this as local triage only, not as the implicit start of a full canon pass.

## Route to a dedicated canon ops lane, agent, or session

Route `canon_doctor` or `canon_fix` through a dedicated canon ops lane, agent,
or runtime session when:
- you need full diagnosis for `source`, `memory`, or `sync`
- you expect preview review, confirm-token handling, or apply decisions
- the work may touch repo-owned canon files or `memory.jsonl`
- findings will need operator judgment, handoff, or tracked follow-through

## Tool mapping

- `canon_status` for local summary and freshness
- `canon_doctor` for diagnosis-first canon investigation
- `canon_fix mode=preview` before any apply
- `canon_fix mode=apply` only from the same reviewed ops flow with the preview token

## Recommended mode mapping

- `status`: run `canon_status` only and return summary, freshness, and whether deeper doctor work is recommended
- `doctor`: run `canon_status` plus `canon_doctor` for `source`, `memory`, and `sync`; do not apply fixes
- `preview-fix`: run doctor first, then preview only safe `memory` and `sync` fixes
- `apply-safe-fix`: run doctor or preview if stale, then apply only safe derived-layer fixes for `memory` and `sync`
- `source-preflight`: run source-focused diagnosis before any durable fix that would touch generated, copied, installed, or runtime surfaces

## Boundary rules

- This skill does not imply detached execution support in the package runtime.
- A dedicated canon ops lane, agent, or runtime session is an operator
  workflow boundary, not a new tool surface shipped by this package.
- In current OpenClaw-style runtimes, a dedicated agent is the natural
  configuration surface; named session keys are usually generated by runtime
  policy rather than declared here.
- `docs/PLUGIN_PACKAGE_CANON.md` and `memory.jsonl` are authority inputs only
  under the detected or explicitly configured canon repo root.
- Installed plugin copies, active skill copies, and workspace notes are never
  authority-side fallbacks for source or memory canon.
- Keep source drift diagnosis-first.
- Keep fixes bounded to shipped `memory` and `sync` scopes only.

If the target repo does not ship these repo-owned canon files, runtime should
return typed prerequisite findings rather than imply generic canon coverage.

## Output expectations

After acting, return:
1. chosen operating mode: local summary or dedicated ops lane/session
2. scope and tool used
3. preview versus apply status
4. files or canon surfaces reviewed or changed

## Report contract

Prefer a fixed structured result with:
- `mode`
- `status`
- `canonicalRoots`
- `derivedLayers`
- `findings`
- `safeFixes`
- `blockedActions`
- `confidence`
- `recommendedNextStep`
